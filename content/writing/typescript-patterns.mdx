# TypeScript Patterns I Use Every Day

After years of writing TypeScript, I've developed a set of patterns that I reach for constantly. Here are the ones I find most useful.

## 1. Discriminated Unions for State Management

Instead of using booleans to track state, use discriminated unions:

```typescript
// ❌ Avoid this
interface DataState {
  isLoading: boolean;
  isError: boolean;
  data: User[] | null;
  error: Error | null;
}

// ✅ Use this instead
type DataState =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: User[] }
  | { status: "error"; error: Error };
```

This makes impossible states unrepresentable. You can't accidentally have both `isLoading: true` and `isError: true`.

## 2. The `satisfies` Operator

The `satisfies` operator lets you validate that an expression matches a type without widening it:

```typescript
const routes = {
  home: "/",
  about: "/about",
  writing: "/writing",
} satisfies Record<string, string>;

// Type is still { home: "/", about: "/about", writing: "/writing" }
// Not Record<string, string>
```

## 3. Template Literal Types

These are incredibly powerful for building type-safe APIs:

```typescript
type EventName = `on${Capitalize<string>}`;
type CSSProperty = `--${string}`;

function setCustomProperty(name: CSSProperty, value: string) {
  document.documentElement.style.setProperty(name, value);
}

setCustomProperty("--primary-color", "#3b82f6"); // ✅
setCustomProperty("primary-color", "#3b82f6"); // ❌ Type error
```

## 4. Branded Types

When you want to differentiate between types that have the same underlying structure:

```typescript
type UserId = string & { readonly __brand: "UserId" };
type PostId = string & { readonly __brand: "PostId" };

function getUser(id: UserId) { /* ... */ }
function getPost(id: PostId) { /* ... */ }

const userId = "abc-123" as UserId;
const postId = "def-456" as PostId;

getUser(userId); // ✅
getUser(postId); // ❌ Type error
```

## 5. `const` Assertions with `as const`

Lock down your objects and arrays to their literal types:

```typescript
const THEMES = ["light", "dark", "system"] as const;
type Theme = (typeof THEMES)[number]; // "light" | "dark" | "system"
```

---

These patterns have saved me countless hours of debugging. The key insight is that **TypeScript's type system is most valuable when it prevents bugs at compile time**, not just when it provides autocomplete.
